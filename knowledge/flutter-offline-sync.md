# Flutter 离线优先与数据同步(向量库优质语料)

> 用途:为"离线同步/本地数据库/冲突解决"类需求提供检索接地语料。来源见 REFERENCES §25。

## 1. 本地存储选型

- **键值/小配置**:`shared_preferences`(平台原生 prefs 封装,不适合大数据与查询)。
- **结构化关系数据**:`drift`(SQLite 之上的类型安全 ORM,编译期校验 SQL,支持
  响应式查询流)——离线优先应用的主力选择。
- **文档型/高写入吞吐**:`isar` / `hive` 系——无 SQL,适合缓存型数据;注意维护
  活跃度与平台支持随版本变化,选型时核对仓库近期提交。
- **端到端同步托管**:Firebase Firestore 自带离线持久化与自动同步(冲突按
  last-write-wins),适合不想自建同步协议的场景;代价是数据模型与查询能力受限。

## 2. 自建同步协议的要点

- **增量队列**:本地写操作进 outbox 队列(单调递增序号 + 幂等 id),联网后按序上传;
  上传必须幂等(服务端按操作 id 去重),否则重试会产生重复数据。
- **冲突解决三档**:last-write-wins(最简单,会丢更新)→ 字段级合并(大多数表单
  场景够用)→ CRDT(协同编辑才需要,复杂度高一个量级)。先问"两个设备同时改同
  一条记录的概率",再决定档位。
- **时钟不可信**:移动设备时钟可被用户修改,排序用服务端授时或混合逻辑时钟(HLC),
  不要用客户端 wall clock 做冲突仲裁。
- **同步状态显式化**:每条记录带 `pending/synced/conflict` 状态,UI 可见——
  静默丢失同步失败是离线应用最难排查的 bug 来源。

## 3. 连接检测

`connectivity_plus` 只报告网络接口状态,不等于"能到达你的服务器"——判断在线
应以一次轻量心跳请求为准,接口状态仅用于提前降级 UI。
