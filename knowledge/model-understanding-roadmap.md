## 第一段:必要(影响当下决策,优先做)

### [x] 1. A4 自建评测方法论
- **产出**:`knowledge/claude-eval-methodology.md`(样本构造细则、model × effort 扫描矩阵、

  样本量与统计方法;质量-成本前沿的画法与读法。
- **为什么必要**:两份文档反复得出同一个结论——"自建评测集是选型的唯一可靠依据"。

- **验收**:给出一个可直接套用的评测模板(任务样本格式、打分规则、两轴扫描脚本伪代码、
  结果解读示例),读者照做就能为自己的负载得出 model/effort 结论。

### [ ] 2. A2 API 实战手册
- **是什么**:adaptive thinking + effort + prompt caching + batch 的完整调用示例集
  (curl/Python),含成本测算表、重试/超时/降级模板、回退事件的日志埋点示例。
- **为什么必要**:是把文档结论变成代码的最后一公里;§8 工程集成要点目前只有原则没有样例。
- **验收**:每个参数组合有可运行示例;给出"1000 次调用在不同 effort/缓存命中率下的成本表"。
- **细化(v3)**:成本表的 token 长度必须用第 1 条评测集的实测分布,不得拍脑袋;
  回退埋点直接复用 `claude-fallback-playbook.md` §2 的三字段最低要求。

### [x] 3. A3 安全分类器与回退的深度分析
- **产出**:`knowledge/claude-fallback-playbook.md`(检测清单、误伤处理决策流、
  回退感知的归因顺序、SLI/告警阈值、应用层守门员模式照搬细则)。
- **是什么**:分类器机制推断、误伤边界实测方法、双形态发布的产业影响、
  应用层照搬"强模型+独立守门员+可接受回退体"结构的设计指南。
- **为什么必要**:回退是 Fable 5 接入方唯一不可绕过的运行时不确定性(§9 失效模式第一条);
  不理解它,质量抖动无法归因,生产事故无法定位。
- **验收**:给出"判断本次响应是否被回退"的检测清单 + 被误伤时的改写/降级决策流程图。

### [ ] 4. 竞品横向对比与多厂商选型
- **是什么**:Fable 5 / Opus 4.8 与同代 OpenAI / Google / DeepSeek / 开源系(Llama/Qwen)
  的能力-价格-生态三轴对比;什么负载不该用 Claude;多厂商冗余架构(避免单点依赖)。
- **为什么必要**:现有文档只在 Anthropic 内部选型,是个盲区——真实决策第一问是"用谁家",
  第二问才是"用该家哪档"。本仓库自身就接 DeepSeek,这条直接影响项目选型。
- **验收**:三轴对比表 + "什么信号出现时应重新评估厂商选择"的触发清单。
- **细化(v3)**:对比必须跑在第 1 条的同一评测集上(方法已就绪,只差扫描他厂矩阵);
  触发清单与 `prediction-tracker.md` 的回填触发条件合并维护,避免两处过期。

### [ ] 5. 成本工程专题
- **是什么**:token 经济学全景——输入/输出/思考 token 的成本结构,缓存命中率建模,
  降本手段优先级(缓存 > 路由降级 > effort 降档 > prompt 压缩 > batch),预算告警设计。
- **为什么必要**:"单价高 ≠ 总成本高"只是口号,没有成本模型就无法验证;成本失控是
  生产 LLM 应用最常见的下线原因之一。
- **验收**:可套用的成本测算表(参数:调用量/缓存命中率/effort 分布)+ 降本决策顺序图。
- **细化(v3)**:降本动作的前提是 `claude-eval-methodology.md` §2.4 的前沿图——
  任何降本(降档/降路由)必须给出"质量损失 ≤ 评测噪声"的证据,否则只是转移成本到事故。

### [ ] 6. 结构化输出与工具调用可靠性
- **是什么**:JSON/schema 约束输出的失败模式与修复策略(本仓库的 JSON 自修复是个雏形);
  工具调用的幂等设计、参数幻觉防御、并行调用的一致性问题。
- **为什么必要**:Agent 系统的可靠性上限 = 结构化接口的可靠性;这是 Cognition 表扬
  Opus 4.8 "工具调用干净"的反面——干净不是默认,是要工程出来的。
- **验收**:失败模式清单(含复现条件)+ 每种模式的防御/修复模板。

### [ ] 7. 上下文工程:RAG vs 长上下文 vs 记忆位
- **是什么**:1M 上下文时代三种供给信息的方式各自的成本/时效/可控性边界;
  "lost in the middle"类注意力衰减的实证现状;本仓库 skill 注入 vs 向量检索的取舍就是同一问题。
- **为什么必要**:这是每个 LLM 应用架构的第一个分叉口,选错了后面全是补丁;
  且 prompt caching 的出现改变了传统 RAG-vs-长上下文的答案。
- **验收**:三方式决策矩阵(语料规模 × 更新频率 × 查询分布)+ 混合架构的参考形状。

### [ ] 8. 模型迁移与退役策略
- **是什么**:新版本发布/旧版本退役时的平滑迁移流程:回归评测门禁、双跑对比期、
  prompt 的版本适配(同 prompt 在不同代模型上行为偏移),回滚预案。
- **为什么必要**:Opus 线 6 个月迭代 4 版——迁移不是偶发事件而是常态运营;
  没有流程的团队每次发布都在裸套升级。
- **验收**:一份可复用的迁移 checklist(触发→评测→双跑→切流→回滚条件)。

### [x] 8.5 能力固化与持续强化框架(v4 新增)
- **产出**:`knowledge/capability-fixation.md`(棘轮机制:测量/门禁/沉淀;四条退化理论依据;
  无梯度固化的分层存放细则;验证权力分工;强化飞轮设计)与
  `knowledge/capability-degradation-taxonomy.md`(八种退化源的检测/处置细则)。

- **是什么**:回答"如何让系统能力只升不降"的总框架,是第 1/3/8/14 条的理论顶层。
- **后续(v5 更新)**:飞轮①②④步已全部落码(`harvest_failures`/`triage_candidates`+`eval_store`/`eval_gate`),理论→代码映射见 capability-fixation §9;③修是人/模型协作环节,无独立代码。

### [~] 8.6 本仓库产出质量评测集(v4 新增,v5 转进行中)
- **是什么**:按 `claude-eval-methodology.md` 的模板,从 runs.jsonl 与收割候选

## 第二段:长期(持续增值,尽早开账、慢慢养)

### [x] 9. B5 可证伪预测的跟踪表

### [ ] 10. B1 瓶颈轮换史完整复盘
- **是什么**:2017→2026 逐年证据链:每次瓶颈转移的标志性论文/产品/价格变化,
  验证"瓶颈预测法"在历史上的命中率。
- **为什么长期**:是 §8 瓶颈轮换论的实证基础;证据收集量大,且新发布会持续追加章节。

### [ ] 11. B2 "下一个瓶颈"论证
- **是什么**:非可判定领域反馈信号(品味/设计/研究方向感)为什么难;
  现有尝试(成对比较、宪法扩展、专家蒸馏、过程奖励泛化)各自的天花板与残余空间。
- **为什么长期**:这是框架里置信度最低、赌注最大的判断,需要随证据滚动修订;
  一旦被证实/证伪,直接改写第一段所有选型建议的有效期。
- **验收**:对每条现有路线给出"它能走到哪、卡在哪"的机制分析 + 可观察的领先指标。

### [ ] 12. A5 记忆工程专题
- **是什么**:文件记忆/笔记自改进的机制分析;应用层"记忆位"的设计模式
  (schema、读写时机、压缩策略、与 prompt caching 的相互作用)。
- **为什么长期**:预测 3 说记忆收益会持续拉大,这个专题的价值随之复利;
  但目前公开证据只有 Slay the Spire 一例,需等更多一手材料再写深。
- **验收**:至少 3 种记忆位设计模式,各带适用场景与反模式。

### [ ] 13. 多 Agent 编排专题
- **是什么**:何时拆多 Agent、何时单 Agent 更强(强模型 + 少脚手架的趋势正在吞噬
  多 Agent 的理由);编排模式(主从/流水线/辩论/评审)的适用边界与失败模式。
- **为什么长期**:Fable 5 的"少脚手架"趋势正在重写这题的答案,现在写死结论会快速过时;
  需跟随证据滚动维护。
- **验收**:"拆不拆"决策树 + 每种编排模式的失败模式清单。

### [~] 14. 生产可观测性与在线评测(v5 转进行中)
- **是什么**:LLM 应用的 SLI/SLO 设计(质量、成本、延迟、回退率)、线上采样回评、

- **为什么长期**:依赖第 1 条先落地;且指标体系要在真实流量里养出来,无法一次写完。
- **验收**:指标字典(含采集点)+ 质量回归归因决策树(先查回退→再查缓存→再查模型版本)。
- **细化(v4)**:归因决策树已有完整雏形——`capability-degradation-taxonomy.md` 的八退化源总顺序

  `src/flutter_agent/degradation.py`(D1 模型变更/D1-D2 失败率漂移/D5 缓存代理/D3 语料哈希 diff)
  + `scripts/degradation_report.py`(一键巡检,exit 0/1)+ `scripts/corpus_drift.py`

### [ ] 15. 幻觉与接地(grounding)专题
- **是什么**:幻觉的机制分类(参数记忆失真/推理跳步/迎合性编造);引用强制、
  检索接地、自检门禁各自能压住哪类;为什么"验证闭环"是唯一通解。
- **为什么长期**:机制理解依赖可解释性研究的进展(transformer-circuits 线),证据在持续更新。
- **验收**:幻觉分类 × 压制手段的有效性矩阵,每格带证据或【推断】标注。

### [ ] 16. 微调 vs 提示工程 vs 蒸馏小模型
- **是什么**:三条定制路线的成本/收益/维护负担对比;什么信号表明该从提示工程
  升级到微调/蒸馏(量大、延迟敏感、领域封闭);与 binglong 师生蒸馏的对照。
- **为什么长期**:前沿模型迭代速度目前压制了微调的性价比,但这个平衡点随价格曲线移动,
  需定期重算。
- **验收**:三路线决策矩阵 + "升级信号"清单。

### [ ] 21. 隐私、合规与数据治理

- **为什么在后**:对个人/小团队项目不是当前瓶颈;一旦面向企业客户会跳升为必要段。
- **验收**:接入前合规 checklist + 脱敏模式清单。
