## 第一段:必要(影响当下决策,优先做)

### [ ] 1. A4 自建评测方法论
- **是什么**:如何为自己的业务构建评测集;model × effort 两轴扫描的实验设计、
  样本量与统计方法;质量-成本前沿的画法与读法。
- **为什么必要**:两份文档反复得出同一个结论——"自建评测集是选型的唯一可靠依据"。
  这是所有其他结论的前置依赖:没有它,§6.1 的 effort 规则和 §7.1 的决策树都只能按直觉执行。
- **验收**:给出一个可直接套用的评测模板(任务样本格式、打分规则、两轴扫描脚本伪代码、
  结果解读示例),读者照做就能为自己的负载得出 model/effort 结论。

### [ ] 2. A2 API 实战手册
- **是什么**:adaptive thinking + effort + prompt caching + batch 的完整调用示例集
  (curl/Python),含成本测算表、重试/超时/降级模板、回退事件的日志埋点示例。
- **为什么必要**:是把文档结论变成代码的最后一公里;§8 工程集成要点目前只有原则没有样例。
- **验收**:每个参数组合有可运行示例;给出"1000 次调用在不同 effort/缓存命中率下的成本表"。

### [ ] 3. A3 安全分类器与回退的深度分析
- **是什么**:分类器机制推断、误伤边界实测方法、双形态发布的产业影响、
  应用层照搬"强模型+独立守门员+可接受回退体"结构的设计指南。
- **为什么必要**:回退是 Fable 5 接入方唯一不可绕过的运行时不确定性(§9 失效模式第一条);
  不理解它,质量抖动无法归因,生产事故无法定位。
- **验收**:给出"判断本次响应是否被回退"的检测清单 + 被误伤时的改写/降级决策流程图。

## 第二段:长期(持续增值,尽早开账、慢慢养)

### [ ] 4. B5 可证伪预测的跟踪表
- **是什么**:把 model-capability-evolution.md §10 的 5 条预测写成带检验日期、
  判据、当前证据栏的跟踪表;每次大模型发布后回填打分。
- **为什么长期**:价值在于时间——预测只有被回看才有意义;表本身一小时就能建,
  但要随每次发布维护。这是"框架是否值得信"的唯一客观检验。
- **验收**:表建立 + 首次回填(以 Fable 5 发布为第 0 个数据点);约定回填触发条件。

### [ ] 5. B1 瓶颈轮换史完整复盘
- **是什么**:2017→2026 逐年证据链:每次瓶颈转移的标志性论文/产品/价格变化,
  验证"瓶颈预测法"在历史上的命中率。
- **为什么长期**:是 §8 瓶颈轮换论的实证基础;证据收集量大,且新发布会持续追加章节。
- **验收**:逐年表格(年份/有效瓶颈/标志事件/转移信号),并对 §8 的四段划分给出修正或确认。

### [ ] 6. B2 "下一个瓶颈"论证
- **是什么**:非可判定领域反馈信号(品味/设计/研究方向感)为什么难;
  现有尝试(成对比较、宪法扩展、专家蒸馏、过程奖励泛化)各自的天花板与残余空间。
- **为什么长期**:这是框架里置信度最低、赌注最大的判断,需要随证据滚动修订;
  一旦被证实/证伪,直接改写第一段所有选型建议的有效期。
- **验收**:对每条现有路线给出"它能走到哪、卡在哪"的机制分析 + 可观察的领先指标。

### [ ] 7. A5 记忆工程专题
- **是什么**:文件记忆/笔记自改进的机制分析;应用层"记忆位"的设计模式
  (schema、读写时机、压缩策略、与 prompt caching 的相互作用)。
- **为什么长期**:预测 3 说记忆收益会持续拉大,这个专题的价值随之复利;
  但目前公开证据只有 Slay the Spire 一例,需等更多一手材料再写深。
- **验收**:至少 3 种记忆位设计模式,各带适用场景与反模式。
