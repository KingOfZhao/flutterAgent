# 上下文工程决策矩阵:skill 注入 vs 向量检索 vs 长上下文

> 性质:路线图第 7 条的落地产出。三种"把知识递给模型"的方式不是竞争关系,
> 是不同坐标下的最优解;本仓库同时用了前两种,是现成的对照实验体。
> 时点:2026-06-10。理论锚:lost in the middle(arxiv 2307.03172)/ prompt caching 改变成本结构。

## 1. 三方式的本质差异

| 维度 | skill 注入(策展语料) | 向量检索(RAG) | 长上下文全量塞入 |
|---|---|---|---|
| 知识形态 | 纪律/方法论(怎么做) | 事实/片段(是什么) | 原始材料(未加工) |
| 进入门槛 | 人工策展,慢而准 | 自动入库,快而糙 | 零加工 |
| 错误模式 | 语料漂移(D3) | 检索污染(D8)+ 错配 | 中部丢失(D4)+ 成本爆炸 |
| 质量上界 | 高(每条都被人判过) | 中(受 embedding 质量限) | 取决于模型长上下文能力 |
| 缓存友好度 | 高(稳定前缀可全缓存) | 低(每查询动态拼装) | 中(材料稳定则可缓存) |

## 2. 决策矩阵(语料规模 × 更新频率 × 查询分布)

- **小语料(<100 条)+ 低频更新 + 查询集中** → skill 注入。
  本仓库 61 个 skill 即此象限:全量排序+预算截断比检索更稳,且前缀可缓存。
- **大语料(>1k 条)+ 高频更新 + 查询长尾** → RAG。
  策展跟不上更新速度时,接受 D8 风险换覆盖,但出处字段规格(taxonomy D8)不可省。
- **材料一次性使用(单次分析任务)** → 长上下文直塞。
  为一次性任务建索引是浪费;但 D4 细则适用:关键指令放头尾,材料按相关性排序。
- **混合形状(本仓库的演化方向)**:skill 注入为骨架(纪律层,稳定前缀),
  RAG 补事实层(ingest 脚手架成熟后),长上下文只用于单次任务——三层互不替代。

## 3. 与能力固化的接合点(为什么这是个固化问题)

三方式的真正差异在**知识的可门禁性**:

- skill 语料在 git 里,有哈希审计(`corpus_drift.py`)、改动可走产出门禁——**可固化**;
- RAG 库内容不经 review 进入生产,固化要靠入库校验与来源审计——**半可固化**;
- 长上下文是用后即焚,无任何沉淀——**不可固化**。

由此推论:**能力要沉淀的知识必须最终落到 skill 层**。RAG/长上下文里被反复用到的内容,
就是"待策展进 skill"的信号(使用频次即策展优先级)——这是飞轮①筛在知识侧的镜像。

## 4. 切换信号

| 当前方式 | 升级/切换信号 |
|---|---|
| skill 注入 | 策展积压 > 1 个月 或 语料超预算被频繁截断(ranker 命中预算上限占比 >30%) |
| RAG | 检索错配成为失败直方图首因;或语料缩减到可策展规模 → 回退 skill 注入 |
| 长上下文 | 同类材料三次以上被复用 → 入库(RAG)或策展(skill) |

## 诚实边界

- 表中"质量上界"排序是机制推断+行业经验,本仓库未做三方式同负载对照实测。
- prompt caching 的成本优势数据随上游定价变动,引用时核对当期价目。
- "查询分布"判断需要任务类型标注(D7 同款依赖),标注缺位时本矩阵只能粗用。
