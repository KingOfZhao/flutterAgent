# 模型迁移 checklist:触发→评测→双跑→切流→回滚

> 性质:路线图第 8 条的落地产出。把"迁移不是事件而是常态运营"变成一张可逐项勾选的单子。
> 理论底座:`capability-fixation.md` §2(棘轮)/ taxonomy D1(版本更新是八退化源之首)。
> 时点:2026-06-10;以本仓库(DeepSeek 上游 + 计划中的 Claude 路由)为对象,但形状可复用。

## 0. 迁移的本质判断

迁移 = 一次**全量的 D1 退化事件,只是有公告**。所有 D1 细则在此适用,区别仅在于
你有时间窗口提前准备。由此推出本 checklist 的硬规则:
**没有产出门禁(8.6)的迁移是裸套升级,任何"看起来没问题"都不构成证据。**

## 1. 触发(以下任一即启动)

- [ ] 上游发布新版本公告 / 宣布旧版本退役时间表;
- [ ] `scripts/degradation_report.py` 的 D1 检测器报出 `stage.model` 变更转移点
  (上游静默切换——比公告更危险,立即进入第 2 步);
- [ ] 价格变动幅度 > 20%(成本结构变化也可能值得迁移);
- [ ] 内部评测显示新模型在本负载上的前沿点显著占优(`claude-eval-methodology.md` §2.4)。

## 2. 评测(切流前必须全绿)

- [ ] 锁定当前版本基线:在 working 集跑 `scripts/eval_gate.py` 留存 `results_baseline.jsonl`;
- [ ] 新版本同集跑分,门禁判定 ≥ tie(不劣于);hard 回归一票否决;
- [ ] 检查未文档化行为依赖:格式容忍、隐式假设(D1 细则——版本间最先碎的);
- [ ] 成本重测:同样本集的 token 用量对比(单价 × 用量才是成本,二者都会变)。

## 3. 双跑对比期(建议 ≥ 1 周或 ≥ 100 真实请求)

- [ ] 影子流量:新版本旁路跑真实请求,不返回用户;
- [ ] 对比维度:失败原因直方图(`scripts/harvest_failures.py` 的 reason histogram)、
  质量分布、延迟 P95、token 成本;
- [ ] 双跑期收割的失败样本照常走 triage——**迁移期是评测集最肥沃的采样窗口**
  (两个版本的行为差异本身就是高信息样本)。

## 4. 切流

- [ ] 灰度递进:5% → 25% → 100%,每级停留期内 D1/D2 失败率漂移检测无告警;
- [ ] 缓存主动失效(D5:缓存键含 prompt/模型版本,确保不混血);
- [ ] 100% 后保留旧版本可用配置 ≥ 2 周(回滚通道不拆)。

## 5. 回滚条件(事先写死,不许临场争论)

- [ ] 门禁 verdict 转 blocked(working 集重跑);
- [ ] 失败率漂移检测告警且归因到新版本(排除 D2/D5 后);
- [ ] 用户侧硬投诉 ≥ 2 起且复现归因到新版本。
  任一命中 → 立即回滚,事后把触发样本入评测集再重新走第 2 步。

## 与既有工具的对应

| 步骤 | 已落码工具 | 缺口 |
|---|---|---|
| 触发-静默切换 | `degradation.detect_model_change` | — |
| 评测-门禁 | `eval_gate` + `eval_store` | 真实 regression 样本(8.6 剩余) |
| 双跑-失败收割 | `harvest_failures --since` | 影子流量开关需在服务层实现 |
| 切流-漂移监控 | `detect_failure_rate_shift` | 灰度分流需在服务层实现 |
| 回滚 | — | 版本钉住依赖上游支持 |

## 诚实边界

- 本 checklist 未经一次真实迁移检验;首次执行后必须回填修订(预测 6 的判据即此)。
- 灰度比例与停留期是先验,不是标定值;小流量服务可简化为 5%→100% 两级。
- 影子流量与灰度分流在本仓库服务层尚无实现——这两项是 checklist 的执行前提缺口。
