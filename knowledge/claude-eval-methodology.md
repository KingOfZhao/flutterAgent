# 自建评测方法论:model × effort 两轴扫描实操手册

> 性质:方法论 + 可直接套用的模板。对应 `model-understanding-roadmap.md` 第 1 条(A4)。
> 前置阅读:`claude-fable5-opus48.md` §6/§7、`model-capability-evolution.md` §7。
> 时点:2026-06-10。

## 0. 为什么这份文档是所有选型结论的前置依赖

两份主文档反复得出同一结论:公开基准对这一代模型区分度下降,**自建评测集是选型的唯一可靠依据**。
没有它,"默认 high、评测确认不掉分再降档"只是按直觉执行的口号。本文把口号变成流程。

## 1. 评测集的构造细则

### 1.1 样本来源(按优先级)

1. **生产真实任务的脱敏样本**:过去 30-90 天的真实请求,按任务类型分层抽样;
2. **历史事故样本**:曾经出错/被投诉的任务,天然是高区分度样本;
3. **边界合成样本**:在真实样本上做受控扰动(更长输入、缺失字段、矛盾指令);
4. 最后才是公开基准题——只用于和外部结果对齐,**不进入选型打分**。

### 1.2 样本量与分层

- 每个任务类型 ≥ 30 条(单类型 < 30 时合并到邻近类型,不要硬撑统计);
- 总量起步 100-300 条即可:目的不是发论文,是把"质量-成本前沿"画出来;
- 必须保留 20% 作为**封存集(holdout)**:调 prompt 时不许看,只在最终选型时跑一次。
  评测集被 prompt 迭代"过拟合"是自建评测最常见的失效方式。

### 1.3 样本格式(可直接套用)

```jsonc
{
  "id": "task-0042",
  "type": "refine",                 // 任务类型,用于分层统计
  "input": { ... },                 // 完整请求(含 system prompt 版本号)
  "rubric": {
    "pass_criteria": [ ... ],       // 机器可判定的硬性条款(JSON schema 通过/引用真实/编译通过)
    "quality_dims": [               // 软性维度,1-5 分
      {"dim": "completeness", "desc": "..."},
      {"dim": "groundedness", "desc": "引用是否可溯源"}
    ]
  },
  "gold": { ... }                   // 可选:参考答案(存在时启用自动比对)
}
```

### 1.4 打分规则细则

- **硬性条款一票否决**:任何 pass_criteria 失败 → 该样本记 0 分,软性分不再计入。
  理由:生产里 schema 错误不是"扣 1 分",是事故。
- 软性维度用 **LLM-as-judge + 抽样人工校准**:
  - judge 模型必须 ≠ 被评模型(避免自我偏好,已有多项研究证实同族模型互评偏高);
  - 每轮评测随机抽 10% 由人复核,judge-人一致率 < 80% 时该维度作废重写 rubric;
  - judge 的 temperature=0、prompt 固定版本化,否则分数不可比。
- 最终分 = pass_rate × mean(quality)。两个数都要报告,不要只报乘积。

## 2. model × effort 两轴扫描

### 2.1 实验矩阵

| | low | medium | high | xhigh | max |
|---|---|---|---|---|---|
| Opus 4.8 | ○ | ● | ● | ● | — |
| Fable 5 | — | ● | ● | ● | ● |

●= 必扫;○= 可选;—= 不可用或不值得(Fable 5 用 low 违背其定位;Opus 4.8 不支持 max,见主文档 §6)。
每个格子跑全量评测集,`temperature=0`,同一 prompt 版本。

### 2.2 每格记录的指标

```
score(质量分) | pass_rate | input_tokens | output_tokens(含思考) | cost_usd | p50/p95 延迟 | 回退次数(仅 Fable 5)
```

回退次数单独记:被回退的样本实际是 Opus 4.8 在答,混入 Fable 5 格子会污染两边的结论
(细则见 `claude-fallback-playbook.md`)。

### 2.3 扫描脚本伪代码

```python
for model, effort in matrix:
    for sample in eval_set:
        resp = call(model=model, effort=effort, temperature=0, **sample.input)
        fallback = detect_fallback(resp)          # Fable 5 响应中会告知
        score = judge(resp, sample.rubric)        # 硬性先判,软性后评
        log_row(model, effort, sample.id, score, resp.usage, fallback)
aggregate(by=["model", "effort", "type"])         # 必须按任务类型分层看
```

### 2.4 质量-成本前沿的画法与读法

- 横轴:每千次调用成本(用实测 token,不要用标称价×拍脑袋长度);纵轴:质量分。
- 把 8-9 个格子画成散点,**只连接帕累托前沿**(没有任何点在其右上方的点)。
- 读法细则:
  1. 不在前沿上的格子直接淘汰,无论单项多好看;
  2. 前沿上相邻两点之间看**边际成本/边际质量**:每 +1 分质量要多花多少钱,贵不贵由错误代价定;
  3. 按任务类型分层重画:全局前沿掩盖"某类型只有 Fable 5 能及格"这类关键事实——
     **分层结论优先于全局结论**,这正是双模型路由(主文档 §8.1)的数据来源;
  4. 两条曲线交叉(某 effort 下便宜模型反超)是路由阈值的天然标定点。

### 2.5 统计细则

- 同格子重复 3 次(即使 temperature=0,思考采样与上游波动仍引入方差),报均值±极差;
- 两格子差异 < 评测集自身噪声(用重复跑的极差估)时,**判为平手,选便宜的**;
- 样本量 100-300 时不要做花哨检验,bootstrap 置信区间足够;区间重叠即平手。

## 3. 结果解读示例

某次扫描(示意数据):

| 格子 | 质量 | $/1k 次 | 结论 |
|---|---|---|---|
| Opus 4.8 + medium | 3.6 | $14 | 前沿点 |
| Opus 4.8 + high | 3.9 | $22 | 前沿点 |
| Opus 4.8 + xhigh | 3.9 | $31 | 被 high 支配,淘汰 |
| Fable 5 + medium | 4.1 | $35 | 前沿点 |
| Fable 5 + xhigh | 4.4 | $61 | 前沿点 |

解读:xhigh 对 Opus 4.8 无增益 → 该负载已到 Opus 能力顶;Fable 5 medium 即超 Opus 全档 →
"token 效率即真实成本"(主文档 §6)在此负载成立;路由策略 = 常规走 Opus 4.8 high,
分层里不及格的类型走 Fable 5 xhigh。

## 4. 评测的生命周期

- **触发重扫**:模型新版本发布 / prompt 大改 / 任务分布漂移(新类型占比 >10%)/ 价格变动;
- 每次重扫产出一行决策记录:`日期 | 矩阵 | 前沿 | 路由决定 | 下次触发条件`;
- 评测集每季度补充生产新样本并淘汰过时样本,封存集同步轮换;
- 与在线指标闭环:线上质量回归时,第一步是把出问题的样本加入评测集复现,
  而不是直接改 prompt(归因顺序见 `claude-fallback-playbook.md` §4)。

## 诚实边界

- 本文方法适用于"结果可由 rubric 判定"的负载;品味/创意类负载的评测是开放问题
  (`model-capability-evolution.md` §8 判断这正是下一个行业瓶颈)。
- LLM-as-judge 的偏差(冗长偏好、位置偏好、自我偏好)只能缓解不能消除;人工抽检比例不可省。
- 示例数据为示意,非实测;首次实测结果应回填到本文档作为案例。
