# 结构化输出可靠性:JSON 产出的失效模式与防御分层

> 性质:本仓库 pipeline 每个 stage 都依赖模型输出可解析 JSON——结构化输出可靠性
> 是产出能力的地基,地基抖动会被上层(验收覆盖、包校验)放大成整单失败。
> 时点:2026-06-10。与 taxonomy D1/D2/D4 交叉;失效样本是评测集的优质 regression 来源。

## 1. 失效模式分类(按本仓库 runs.jsonl 可观测到的形态)

| # | 失效模式 | 形态 | 主要诱因 |
|---|---|---|---|
| S1 | 围栏污染 | JSON 外包裹 ```json 围栏或解释性文字 | prompt 未明示"只输出 JSON";模型版本风格变化(D1) |
| S2 | 截断 | max_tokens 触顶,JSON 不闭合 | 输出长度估计不足;需求复杂度超预期 |
| S3 | schema 漂移 | 键名/嵌套/类型与约定不符(如 string 化的数组) | prompt 中 schema 描述含糊;版本间行为偏移(D1) |
| S4 | 值域违约 | 结构合法但值非法(空列表、占位符文本、语言错误) | 模型"完成形式不完成实质";回退体接管(D2) |
| S5 | 拒答混入 | 安全/能力拒答文本替代或混入 JSON | 触发上游安全分类器(D2,见 fallback-playbook) |

S1-S3 是**语法层**,机器可全自动判定;S4-S5 是**语义层**,需要 rubric/judge。
这条分界与 `capability-fixation.md` 的"执行验证 vs 定义验证"分界完全重合——
语法层验证可以放心交给代码,语义层验证的定义权必须留在评测集里。

## 2. 防御分层(由便宜到贵,逐层兜底)

1. **约定层**:prompt 末尾的输出契约(只输出 JSON/给出 schema 示例)——
   防 S1/S3 的第一道,也是最脆的一道(版本间最先碎,见 D1 细则)。
2. **解析层**:剥围栏、找首尾花括号、宽容解析——本仓库 stage 解析已做;
   修复式解析的纪律:**修复成功也要记日志**,修复率上升 = S1 恶化的前导指标。
3. **校验层**:pydantic schema 校验(本仓库 `validations`/`stage_valid` 即此)——
   防 S3;校验失败触发重试是便宜的自愈,但重试率必须入监控(掩盖趋势=D6 风险)。
4. **语义层**:验收覆盖检查、包合法性校验(本仓库 `acceptance_gaps`/`PackageValidation`)——
   防 S4;这一层产出的失败就是 `harvest_failures` 收割的四类原因,直通评测集。
5. **门禁层**:S1-S5 任何一类的样本沉淀进评测集后,`eval_gate` 保证同类失效不复发。

## 3. 细则

- 重试预算:语法层失败重试 ≤ 2 次(指数退避不必要——失败原因不随时间变化),
  语义层失败不自动重试(同 prompt 重抽大概率同样失败,浪费 token,应入收割队列)。
- max_tokens 设定:取该 stage 历史 P99 输出长度 × 1.5;S2 出现即说明该系数失准,重标。
- 上游若支持受约束解码(json mode/schema 强制),开启后 S1-S3 趋零,
  但 **S4-S5 不降反升的可能性存在**——形式被锁死后,模型的"不会做"只能表现为值域违约。
  开启前后必须分别留存失败直方图对比(`harvest_failures` 的 reason histogram)。
- 失效样本入库优先级:S5 > S4 > S3 > S2 > S1(语义层失效信息量大、复发代价高)。

## 诚实边界

- S1-S5 占比未在本仓库实测;直方图应在真实流量积累后回填本文档。
- "受约束解码使 S4/S5 上升"是机制推断(【推断】),不是实测结论;开启时按上文 A/B 验证。
- 本文不覆盖流式输出的中途校验——本仓库当前按整段解析,流式是未到的需求。
