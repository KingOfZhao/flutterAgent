# 能力固化论:如何让系统能力只升不降

> 性质:理论框架 + 工程细则。回答一个核心命题:**在模型、prompt、数据、依赖都在持续变化的系统里,
> 如何把已经获得的能力"固定住"——不被任何一次变更弱化,并让每次变更都成为强化的机会。**
> 论点为作者(Devin)的综合判断,理论依据为公开文献(见 §1 与 REFERENCES.md)。时点:2026-06-10。

## 0. 问题的本质:能力是流量不是存量

一个 LLM 应用的"能力"不是写进代码就持久存在的资产。它是六个时变因素的乘积的瞬时值:
模型版本 × prompt 版本 × 知识/skill 语料 × 上下文组装策略 × 上游服务行为(缓存/回退/限流)× 任务分布。
任何一项静默漂移,能力就静默漂移。**所以"固定能力"的本质不是冻结某个组件,
而是建立一种机制,使能力的测量值单调不减**——工程上这叫棘轮(ratchet):允许转动,禁止倒转。

## 1. 理论依据:为什么能力会弱化

四条独立的理论路径,解释了"不做任何坏事,能力照样退化"的原因:

1. **灾难性遗忘(catastrophic forgetting)**:神经网络学新分布时覆盖旧分布的表征
   (McCloskey & Cohen 1989;综述见 https://arxiv.org/abs/1802.07569)。
   对应用层的映射:厂商每次更新模型,都是一次你无法控制的"再训练"——
   同一 prompt 在新版本上行为偏移(prompt 的隐性过拟合被打破)。这是**模型轴**的退化源。
2. **古德哈特定律(Goodhart's law)**:指标成为目标后就不再是好指标
   (形式化分析见 Manheim & Garrabrant 2018, https://arxiv.org/abs/1803.04585)。
   对应用层的映射:对着固定评测集迭代 prompt,会把评测分推高而真实能力不变甚至下降——
   这是**评测轴**的退化源,且最隐蔽,因为仪表盘显示一切向好。
3. **分布漂移(distribution shift)**:训练/标定时的任务分布与当前流量分布脱节
   (经典论述:Quiñonero-Candela et al., *Dataset Shift in Machine Learning*, 2009)。
   对应用层的映射:用户行为演化、新任务类型出现,旧的路由阈值/effort 标定静默失效。
4. **软件熵(software entropy)**:Lehman 演化定律——被使用的系统若不主动维护,复杂度上升、质量下降
   (Lehman 1980)。对应用层的映射:prompt 与 skill 语料的"补丁式增长"使指令间互相矛盾,
   模型在矛盾指令下表现退化——这是**语料轴**的退化源(本仓库上一轮反思发现的 skill 冗余即此类)。

结论:退化不是事故,是默认走向。**不退化本身就需要持续做功**——这是热力学式的论断,
也是为什么"固定能力"必须是机制而不是一次性动作。

## 2. 棘轮机制:能力单调不减的最小充分条件

借鉴的工程原型:NNUE 棋类引擎的权重门禁(新权重必须对旧权重打出统计显著的胜率才允许替换,
SPRT 序贯检验;参见 Stockfish 测试框架 fishtest 的实践)。这个原型之所以值得照搬,
是因为它在一个同样"每次变更都可能退化"的领域里,实现了二十年能力单调上升。

棘轮的三个不可省略的组件:

1. **不可博弈的测量**:评测集(`claude-eval-methodology.md`)+ 封存集轮换 + 生产真实样本持续注入。
   封存集是对抗古德哈特的唯一手段:被迭代看过的样本永久失去门禁资格。
2. **变更门禁(gate)**:任何触及能力六因素的变更(换模型版本、改 system prompt、
   增删 skill、调路由阈值)都必须过门:新配置在封存集上 **不劣于** 当前配置才许合入。
   "不劣于"按 `claude-eval-methodology.md` §2.5 的噪声判据执行——差异小于评测噪声判平手,平手允许通过
   (否则棘轮会卡死,见 §4 僵化风险)。
3. **单向沉淀(pawl,棘爪)**:每次线上失败样本必须走完"复现 → 进评测集 → 修复 → 过门"四步。
   这一步把"偶然踩到的坑"转化为"永久不再踩的坑"——能力的强化正是来自失败的不可逆沉淀,
   这与 Fable 5"自建评测 harness、基于学习自我更新 skills"的官方描述同构:
   **前沿模型在权重层做的事,应用层在语料层同样可以做,而且必须做。**

三者缺一:没有测量,门禁是仪式;没有门禁,沉淀会被下次变更冲掉;没有沉淀,门禁只能防退化、不能强化。

## 3. 为什么"强化"和"防弱化"是同一个机制

直觉上防守(不退化)和进攻(变强)是两件事,实际是棘轮的同一次转动:

- 防弱化 = 门禁拒绝净负向变更;
- 强化 = 沉淀使评测集越来越难 + 门禁只放行"在更难的集合上不劣"的变更。
  评测集难度的单调上升,迫使配置质量单调上升——**强化的速度等于失败样本的沉淀速度**。
- 这解释了 `model-capability-evolution.md` §4 的论断"自治程度的上限 = 自我验证能力的上限"
  在应用层的形态:你的系统能变多强,取决于你把多少真实失败转化成了永久门禁。
- 推论(可检验):两个同样的团队,沉淀流程打通的那个,能力曲线斜率持续为正;
  没打通的那个在每次模型升级时能力跳变(有时向上有时向下),长期看是随机游走。

## 4. 棘轮的反面:僵化风险及其对冲

固化机制自身有两种失败方式,必须设计对冲:

| 失败方式 | 机制 | 对冲 |
|---|---|---|
| 僵化(ossification) | 门禁过严(要求显著优于),所有探索性变更被拒,系统冻结在局部最优 | "不劣于+噪声判平"的放行规则;为探索性变更设独立的灰度通道(小流量双跑),用线上数据补足离线评测的区分度 |
| 评测腐烂(eval rot) | 评测集老化,守护的是昨天的任务分布 | 评测集季度轮换(`claude-eval-methodology.md` §4)+ 任务分布漂移监控(新类型占比 >10% 触发重建) |

僵化比退化更难察觉:退化有事故报警,僵化只有"竞品不知为何越来越好"。
对冲的本质是给棘轮留出受控的"空转区"——探索在灰度区免门禁,转正时补门禁。

## 5. 落到本仓库:flutterAgent 的固化层已有与缺失

对照棘轮三组件盘点(这也是后续迭代的工程方向):

| 组件 | 已有 | 缺失 |
|---|---|---|
| 不可博弈的测量 | 319 个单元测试(守护代码行为) | 守护 **输出质量** 的评测集(测试只验证管道不验证产出好坏) |
| 变更门禁 | CI 测试门禁 + review 闭环 | skill 语料变更的质量门禁(改一个 SKILL.md 不会触发任何质量评测) |
| 单向沉淀 | `logs/runs.jsonl` 已记录全部运行 | 从 runs 里把差产出筛出来回流成评测样本/skill 修订的流程 |

诊断:本仓库的固化层覆盖了"代码能力",尚未覆盖"产出能力"——
runs.jsonl 是已经在付费采集的失败样本池,却没有棘爪把它们沉淀下来。
这是典型的"向量库孤岛"同款问题(能力建了没接进决策链,见 REFLECTION_vector_integration.md):
**采集了学习信号,但学习回路没有闭合。**

## 6. 无梯度固化:skill 语料是能力的持久化格式

第 1-5 节讲机制,这一节回答存储问题:**能力固化之后,放在哪里才不随模型更换而蒸发?**

四种可选的持久化位置,按"对模型版本的鲁棒性"排序:

1. **模型权重(微调)**:固化最彻底,但绑定单一版本——基座一升级,微调资产清零重做;
   且受灾难性遗忘约束(§1.1)。除非领域封闭且量大,否则是最脆的存储层。
2. **prompt/skill 语料(无梯度蒸馏)**:把判断纪律写成明文规则与范例。鲁棒性最高——
   明文对任何模型可读,模型越强、对明文纪律的执行越忠实(指令遵循是各代模型单调改进的轴)。
   **这意味着 skill 资产随模型升级自动增值而非贬值**——与微调资产的方向相反。
3. **评测集与门禁**:不直接注入能力,但定义"能力不许低于什么"——是负向固化(地板),
   与 skill 的正向固化(天花板牵引)互补。
4. **检索语料/记忆位**:固化事实与案例而非纪律;时效性内容(API 参数、价格)只应放这里,
   放进 skill 会把"会过期的事实"和"不过期的纪律"焊死在一起——这是语料轴退化(§1.4)的常见起点。

由此得到固化的**分层存放细则**:
- 纪律(怎么判断、什么顺序、什么红线)→ skill;
- 事实(版本号、价格、参数名)→ 知识库/检索,带时点标注;
- 底线(不许退到哪)→ 评测集 + 门禁;
- 偏好(语气、格式)→ system prompt,版本化管理。
混放是慢性毒药:事实过期会连带让读者怀疑同文件里的纪律,纪律的权威性被时效性内容拖垮。

与前沿模型的呼应:Anthropic 把"蒸馏"列入 Fable 5 受保护类别(`claude-fable5-opus48.md` §5),
证明**老师信号是核心资产**;而 skill 体系就是无需梯度的老师信号固化——
把强者(人或前沿模型)的判断纪律写成可检索、可测试、可门禁的明文。
本仓库 61 个 skill 的"蒸馏五专家心智模型"路线,正是这个论断的工程实现。

## 诚实边界

- §2 的棋类引擎类比成立的前提是"结果可机器判定";本仓库产出(PRD 质量)是半可判定的,
  门禁需要 LLM-as-judge 参与,继承其全部偏差(见 `claude-eval-methodology.md` 诚实边界)。
- §3 的可检验推论尚无本仓库的实证数据;首个数据点应在沉淀流程打通后回填。
- "能力单调不减"是机制目标,不是定理:封存集泄漏、judge 漂移、灰度通道滥用都会破坏单调性;
  机制的强度等于其最弱组件。
