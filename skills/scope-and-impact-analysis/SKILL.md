# 需求范围界定与变更影响分析

> 分工:本 skill 负责动手前的三个判断——**范围(做什么/不做什么)**、
> **更改半径(会碰到哪些文件/模块)**、**关联性(和哪些已有需求互相牵连)**。
> 需求→任务的结构化拆解见 `task-refinement`;改动落地纪律见 `flutter-refactoring`、`flutter-feature-development`。

> 核心原则:**范围是协商出来的契约,不是揣测出来的默契**。
> 范围说不清,估算、拆解、验收全都建立在流沙上。

---

## 1. 需求范围界定(in / out 边界)

### 1.1 三层范围模型

把任何需求拆成三个圈,逐层显式化:

```
┌─ 明示范围 (explicit) ── 用户原话直接要求的
│  ┌─ 隐含范围 (implied) ── 不做就交付不完整的(错误态/空态/权限/持久化)
│  │  ┌─ 越界范围 (out) ── 顺手能做但用户没要的 → 默认不做
```

判定规则:

- **明示**:逐句对照用户原文,每个动词都是一条范围项。原文没有的功能 = 不存在。
- **隐含**:问"不做它,明示范围算交付了吗?"——算 → 越界;不算 → 隐含范围,**列出来并告知**,不是默默做。
  - 典型隐含项:加载/空/错误态、输入校验、并发/重入、已有数据迁移、亮暗主题一致性。
- **越界**:顺手重构、风格统一、"既然打开了这个文件就……" → 全部默认 out。想做就单独提出,由用户决定。

### 1.2 范围声明产物

动手前产出一段范围声明(给用户或写进任务):

```jsonc
{
  "in_scope": ["明示项...", "隐含项(标注 implied)..."],
  "out_of_scope": ["刻意不做的 + 为什么", "..."],
  "assumptions": ["做出的假设,错了会推翻方案的"],
  "open_questions": ["不澄清就无法继续的问题"]
}
```

要求:

- `out_of_scope` 非空——写不出"刻意不做什么",说明范围还没想清楚。
- `assumptions` 只放**会改变方案**的假设;鸡毛蒜皮不列。
- 模糊需求("优化一下"、"做得好看点")必须先转成可证伪的范围项,否则进 `open_questions`。

### 1.3 范围蔓延的刹车

执行中发现新问题时的处理顺序:

1. **挡路吗?** 不修就无法完成 in_scope → 纳入,并告知"范围被迫扩大+原因"。
2. **不挡路但有风险?** → 记录为 follow-up,交付时一并报告,**不顺手修**。
3. **纯属顺手?** → 不做。一次交付只解决一个范围。

## 2. 更改范围(blast radius / 更改半径)

### 2.1 先测量再动手

改动前回答三个问题,用工具取证而不是凭记忆:

1. **直接触点**:实现 in_scope 必须编辑哪些文件?(搜索定位,列出清单)
2. **传播面**:这些文件被谁依赖?——改公共 API/model/theme/路由,引用方全是潜在受影响点(grep 引用、看导出)。
3. **共享状态**:是否碰到全局单例、数据库 schema、持久化格式、缓存 key、平台配置?碰到 = 半径放大一级。

### 2.2 半径分级与对策

| 半径 | 特征 | 对策 |
|------|------|------|
| 局部 | 单文件/单 widget,无导出变更 | 直接改,常规测试 |
| 模块 | 改模块内公共接口,引用方在同模块 | 列出引用方清单,逐一确认编译+行为 |
| 跨模块 | 改共享 model/服务/theme/路由 | 改动前列影响清单给用户;考虑兼容层渐进迁移 |
| 持久层/协议 | 改 DB schema、序列化格式、API 契约 | 必须有迁移方案+回滚路径,否则不动 |

### 2.3 最小更改纪律

- **半径与需求成比例**:一句话需求改出 20 个文件 → 停下来,要么方案错了,要么混入了越界范围。
- **一个 PR 一个意图**:行为变更与重构分开提交;同文件里"顺手改名/排版"也算混入。
- **上游最小修优先**:修 bug 找根因处单点修,不在每个调用点打补丁(下游补丁 = 半径假装很小,实际债务变大)。
- 列出的 `files_touched`(见 `task-refinement` 阶段 3)就是半径承诺,实际超出要说明原因。

## 3. 需求关联性(关联图)

### 3.1 四种关联关系

新需求进来,对照已有需求/功能逐一过四类关系:

| 关系 | 判定问题 | 处理 |
|------|---------|------|
| **依赖** (depends-on) | 它没做完,新需求能开工吗? | 排序:被依赖项先行;`depends_on` 不能成环 |
| **冲突** (conflicts) | 两者对同一行为/数据有矛盾要求吗? | 上抛给用户决策,**不要自行折中** |
| **重叠** (overlaps) | 是否动同一批文件/同一状态? | 合并实现或排好串行顺序,避免互相踩 |
| **连带** (ripple) | 做了它,哪些已有功能的行为会变? | 列入回归测试范围,验收时一并验证 |

### 3.2 关联性排查清单

判定连带影响时按共享资产排查,而非凭感觉:

- **共享数据**:同一 entity/表/缓存——改字段语义,所有读写方都连带。
- **共享 UI 资产**:theme token、公共组件、路由表——改一处,消费方全连带(golden test 是探测器)。
- **共享流程**:登录态、权限、生命周期钩子——新需求插入流程中段,上下游都要回归。
- **共享配置**:env/flavor/平台权限清单——一个平台的需求可能连带另一个平台的构建。

### 3.3 关联性产物

```jsonc
{
  "depends_on": ["F-002 登录(token 可用是前提)"],
  "conflicts": ["与 F-007 的离线缓存策略矛盾:本需求要求实时,待决策"],
  "overlaps": ["与进行中的 T-014 同改 auth_repository.dart,需排序"],
  "ripple": ["改动用户 model 后,个人页/评论区展示逻辑需回归"]
}
```

四个字段都为空也要显式写空——"确认过没有关联"和"没想过"是两回事。

## 4. 三件套如何衔接

```
需求进来
 → §1 范围声明(in/out/assumptions/open_questions)
 → §3 关联图(depends/conflicts/overlaps/ripple)   ← 冲突项先决策
 → §2 半径测量(files_touched + 分级对策)
 → 喂给 task-refinement 做正式拆解(范围项→story,半径→files_touched,连带→回归测试)
```

- 范围变 → 半径和关联图必须重算,不能沿用旧结论。
- 半径远大于范围 → 回头检查范围是否混入越界项。

## 反模式

- ❌ 不写 out_of_scope,默认"用户应该懂我不做什么"。
- ❌ 把隐含范围默默做掉不告知,或反过来把隐含范围(错误态/迁移)当越界砍掉。
- ❌ "既然打开了这个文件就顺手……"——范围蔓延的头号入口。
- ❌ 凭记忆判断影响面,不 grep 引用、不查依赖,改完才发现下游编译炸了。
- ❌ 行为变更和重构混在一个提交,出问题无法二分定位。
- ❌ 发现需求冲突自行折中两边,不上抛决策。
- ❌ 改共享 model/theme 不列消费方清单,连带影响靠用户上线后发现。
- ❌ 关联图四字段不写 = 没排查,把"没想过"伪装成"没有关联"。

## 心智模型与诚实边界

> 配合 `flutter-engineer-mindset`(通用思维底座)与 `flutter-skill-distillation`(女娲蒸馏法)使用。

**镜片(怎么想):**

- **范围是契约**:in/out/assumptions 写出来才存在;没写出来的边界等于没有边界。
- **半径用证据测量**:影响面靠 grep/依赖图取证,不靠记忆和直觉。
- **半径与需求成比例**:小需求大半径是危险信号——先怀疑方案,再怀疑范围。
- **关联是图不是列表**:依赖/冲突/重叠/连带四种边,缺一种就有一类事故没防。
- **冲突上抛,不私下折中**:范围与优先级的取舍是产品决策,不是工程师的善意猜测。

**诚实边界:**

- 范围界定依赖用户澄清:open_questions 没回答时,任何范围声明都是带假设的草案。
- 静态影响分析(grep/依赖)抓不住运行时反射、动态路由、字符串拼 key 等隐式依赖,需测试兜底。
- 本 skill 给判定框架与产物格式,不替用户做"做不做"的取舍。
- 隐含范围的判定有主观成分:同一项在不同团队标准下可能是 implied 也可能是 out,以用户确认为准。
